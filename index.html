We use Domain Scout for our own naming and domain work. This page documents what it does today and how it can fit inside other stacks.

Why this exists

Most domain tools are built to answer “is this one domain free?”, powered by third-party APIs, rate limits and subscriptions. Naming a product or studio is a different problem: you work with lists, patterns and constraints. Domain Scout is built for that layer: predictable runs, transparent logic and outputs you can actually read.

What Domain Scout actually does

Under the hood, Domain Scout combines DNS prefiltering, registry-level checks and a light scoring model:

• bulk scanning of large wordlists or generated names;

• dedicated handlers for .com/.net (Verisign), .it and .ch;

• filters for length, prefixes, suffixes and blacklist words;

• 0–100 score per free domain based on length, TLD value, readability;

• exports as TXT, CSV and Markdown for internal docs and client decks.

Who it is for

Domain Scout is not a consumer product. It is meant for teams that ship things:

• solo founders validating names before design and legal;

• studios and agencies doing naming rounds for clients;

• developers and makers who prefer a script-friendly scanner.

FAQ

Does it use external APIs?

Domain Scout is built to work without per-request API pricing. It leans on DNS prefiltering and registry-level checks, so you can run thousands of candidates without worrying about quotas.

Can we plug in our own wordlists?

Yes. Wordlists are the core of how it works: you bring your own data, apply blacklist rules and, if needed, generate brandable variations on top.

Is there a hosted version?

Today, Domain Scout runs locally or on a VPS. A hosted layer can sit on top of the same engine later, adding UI, automation and team access.
